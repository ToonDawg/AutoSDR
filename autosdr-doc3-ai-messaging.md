## 1. Overview

This document defines the AI layer and messaging pipeline for AutoSDR: how tone is
captured during setup, how outreach messages are generated and evaluated, how incoming
replies are classified, how the connector interface works, and how all of these pieces
compose into the full send/receive loop.

The AI layer is intentionally structured as a sequence of focused, single-purpose
LLM calls rather than one large autonomous agent. This makes each step auditable,
testable, and replaceable independently.

---

## 2. LLM Interface

All LLM calls are made via **LiteLLM**. This provides a single consistent interface
regardless of whether the owner is using a cloud provider (Google, OpenAI,
Anthropic) or a local model via Ollama.

```python
# All calls use this pattern — provider is swapped via config, not code
from litellm import acompletion

response = await acompletion(
    model=settings.llm.model_main,   # e.g. "gemini/gemini-3-flash-preview"
    messages=[...],
    temperature=settings.llm.temperature_main,
)
```

**Four model slots** live in `workspace.settings.llm`, each independently
overridable via environment variables at process start:

| Slot | Default | Purpose | Why this default |
|---|---|---|---|
| `model_main` | `gemini/gemini-3-flash-preview` | Message generation (outreach + replies) | Quality matters most here; main model; temperature 0.7 |
| `model_analysis` | `gemini/gemini-3-flash-preview` | Lead analysis → angle | Reads bulky `raw_data`; benefits from the main model's reasoning |
| `model_eval` | `gemini/gemini-3.1-flash-lite-preview` | Self-evaluation (JSON scoring) | Deterministic structured output; temperature 0.0; runs 1–3x per outreach — stays on the lite free-tier quota |
| `model_classification` | `gemini/gemini-3.1-flash-lite-preview` | Intent classification on inbound replies | Same profile: deterministic JSON; one call per inbound; lite quota |

Splitting the slots means heavy-volume deterministic work (eval, classification)
runs on the lite model's separate quota bucket, preserving the main-model quota
for generation. Free-tier Gemini quotas are per-model, so this materially
increases the number of leads that can be processed per day.

**Env-variable overrides** (read into `settings.llm` at startup):

```
LLM_MODEL_MAIN=gemini/gemini-3-flash-preview
LLM_MODEL_ANALYSIS=gemini/gemini-3-flash-preview
LLM_MODEL_EVAL=gemini/gemini-3.1-flash-lite-preview
LLM_MODEL_CLASSIFICATION=gemini/gemini-3.1-flash-lite-preview
LLM_TEMPERATURE_MAIN=0.7
LLM_TEMPERATURE_EVAL=0.0
GEMINI_API_KEY=...
```

LiteLLM reads `GEMINI_API_KEY` automatically for the `gemini/*` model prefix.
Swapping to OpenAI is `LLM_MODEL_MAIN=openai/gpt-4o-mini` + `OPENAI_API_KEY=...`
with no code changes.

## 3. Tone Calibration

Tone calibration happens once during workspace setup and produces a `tone_prompt`
— a short, dense text segment that is injected into every outreach generation call
as part of the system prompt.

### 3.1 The Swipe Flow

The frontend presents the owner with a series of paired message samples. Each pair
shows two versions of the same message in different tones (e.g. formal vs. casual,
direct vs. warm, brief vs. detailed). The owner swipes right on the one that feels
more like them, left on the one that doesn't.

At least 10 decisions are required before calibration can be submitted. More
decisions produce a more accurate tone prompt.

**Sample pairs are generated by the backend on demand** using the business context
from setup. They are not static templates. This ensures the samples feel relevant
to the owner's actual business.

### 3.2 Tone Prompt Compilation

After the owner completes the swipe flow, the backend sends all decisions to a
single LLM call that synthesises a `tone_prompt`.

**Input:**
```json
{
  "business_context": "...",
  "decisions": [
    {
      "preferred": "Hey, saw you're running an aged care facility in Caboolture — we help places like yours reduce staff turnover. Worth a quick chat?",
      "rejected":  "Dear Facility Manager, I am writing to introduce our staffing solutions service..."
    },
    {
      "preferred": "Quick one — do you handle your own staff scheduling, or is that outsourced?",
      "rejected":  "I wanted to follow up on whether you currently have a staffing management solution in place."
    }
  ]
}
```

**System prompt:**
```
You are analysing a set of paired message decisions to extract the communication
style of a small business owner. Based on which messages they preferred and which
they rejected, write a tone guide of 100–150 words that describes:
- Formality level (casual / semi-formal / formal)
- Sentence length and structure preferences
- Use of humour or warmth
- How they open and close messages
- Any specific phrasing patterns evident in their preferred messages

Write the tone guide as a set of direct instructions, as if coaching a writer.
Do not reference the sample messages directly. Output only the tone guide text.
```

**Output (stored as `workspace.tone_prompt`):**
```
Write in a casual, direct tone. Keep sentences short — one idea per sentence.
Open with a grounded observation about the recipient's situation, not a
formal introduction. Avoid corporate language like "I am writing to" or
"I wanted to follow up". Use contractions naturally. A touch of dry humour
is fine but never forced. Close with a single, low-pressure question rather
than a call to action. Never use bullet points or sign off formally.
```

### 3.3 Tone Prompt Updates

The owner can re-run the swipe flow at any time from workspace settings. A new
`tone_prompt` overwrites the old one. Threads already in progress continue using
the tone prompt that was active when they were created (stored in `thread.metadata`
at thread creation time — see Doc 2 delta note at end of this document).

## 4. Outreach Generation Pipeline

Every first-contact message goes through four sequential steps. Steps 1 and 2 happen
at thread creation time. Steps 3 and 4 happen immediately after.

```
Step 1: Lead Analysis      → extract personalisation angle from raw_data
Step 2: Store angle        → write to thread.angle
Step 3: Message Generation → draft message using angle + tone_prompt + campaign goal
Step 4: Self-Evaluation    → score draft; regenerate if below threshold (max 3 attempts)
        ↓
Step 5: Send via connector
```

### 4.1 Step 1 — Lead Analysis

A focused LLM call that reads the lead's `raw_data` blob and extracts a single
personalisation hook.

**Input:**
- `workspace.business_data` — what we sell and why it matters
- `campaign.goal` — what outcome we are driving toward
- `lead.raw_data` — everything known about this lead (serialised JSON)
- `lead.name`, `lead.category`, `lead.address` — core fields for context

**System prompt:**
```
You are analysing a business lead to find the single strongest personalisation
angle for a cold outreach message.

The sender's business: {business_data}
Outreach goal: {campaign_goal}

Review the lead data below and identify one specific, concrete detail that:
1. Suggests a pain point, opportunity, or context relevant to the sender's offering
2. Would make the recipient think "they actually looked at us"
3. Can be referenced naturally in a short SMS message

Output a single JSON object:
{
  "angle": "2-3 sentence description of the hook and why it is relevant",
  "signal": "the specific data point from the lead that supports this angle"
}

Output only valid JSON. No preamble.
```

**Example output:**
```json
{
  "angle": "Multiple reviews mention chronic understaffing on weekends — residents waiting 40+ minutes for basic care. This is a direct pain point that a staffing solution addresses. The facility manager is likely aware and frustrated by it.",
  "signal": "Review by Gary Land (2 stars): 'Last week my friend waited for 40 minutes to be attended to... families pay a lot of money and are not getting value for money.'"
}
```

If `raw_data` is empty or contains no useful signal, the agent returns a generic
angle based on the lead's category and location. This is not an error — some leads
will simply have less context to work with.

### 4.2 Step 2 — Store Angle

The extracted `angle` string is written to `thread.angle`. It is used in message
generation and in all subsequent AI reply drafts for this thread. It is not
regenerated on follow-up messages unless the owner explicitly triggers a refresh
from the thread panel.

### 4.3 Step 3 — Message Generation

Generates the actual outreach message using the angle, tone prompt, and campaign goal.

**System prompt:**
```
{tone_prompt}

You are writing a short outreach SMS on behalf of a small business owner.

About the sender's business:
{business_data}

Campaign goal: {campaign_goal}

Personalisation angle for this recipient:
{thread.angle}

Recipient:
- Name: {lead.name}
- Category: {lead.category}
- Location: {lead.address}

Write a single SMS message. Requirements:
- Maximum 160 characters (one SMS)
- Must feel personal, not templated
- Must reference or be informed by the personalisation angle
- Must include a soft call to action aligned with the campaign goal
- Must match the tone described above exactly
- Do not introduce yourself by full name or company name in the first message
- Output only the message text. No quotes, no labels, no explanation.
```

**Note on length:** 160 characters is enforced by the self-evaluation step, not
just the prompt. Some models ignore length constraints in prompts. The evaluator
checks this explicitly.

### 4.4 Step 4 — Self-Evaluation

A separate LLM call (using `eval_model` at temperature 0.0) scores the drafted
message against a rubric before it is sent.

**System prompt:**
```
You are evaluating a drafted outreach SMS message before it is sent.

Score the message on each criterion from 0.0 to 1.0:

1. tone_match: Does the message match the tone guide exactly?
   Tone guide: {tone_prompt}

2. personalisation: Does the message reference something specific to this recipient
   rather than feeling like a template?
   Angle used: {thread.angle}

3. goal_alignment: Is the message clearly oriented toward the campaign goal?
   Goal: {campaign_goal}

4. length_valid: Is the message 160 characters or fewer?
   (1.0 if yes, 0.0 if no — no partial credit)

5. naturalness: Would a real person send this? Does it avoid robotic phrasing,
   excessive formality, or hollow superlatives?

Return a JSON object:
{
  "scores": {
    "tone_match": 0.0–1.0,
    "personalisation": 0.0–1.0,
    "goal_alignment": 0.0–1.0,
    "length_valid": 0.0 or 1.0,
    "naturalness": 0.0–1.0
  },
  "overall": 0.0–1.0,   // weighted average
  "pass": true | false,  // true if overall >= threshold AND length_valid == 1.0
  "feedback": "one sentence explaining the biggest weakness if pass is false"
}
```

**Scoring weights:**
| Criterion | Weight |
|---|---|
| tone_match | 0.25 |
| personalisation | 0.30 |
| goal_alignment | 0.20 |
| length_valid | 0.15 |
| naturalness | 0.10 |

**Pass threshold:** `overall >= 0.85` AND `length_valid == 1.0`

**On failure:** The feedback string is appended to the next generation call as an
additional instruction: `"Previous attempt failed evaluation. Feedback: {feedback}.
Rewrite the message addressing this weakness."` Maximum 3 generation attempts (the
cap is `workspace.settings.eval_max_attempts`). If all 3 fail, the thread is moved
to `paused_for_hitl` status with:

```python
thread.status       = 'paused_for_hitl'
thread.hitl_reason  = 'eval_failed_after_3_attempts'
thread.hitl_context = {
    "last_drafts": ["draft_1", "draft_2", "draft_3"],
    "last_scores": [
        {"overall": 0.72, "breakdown": {...}, "feedback": "..."},
        {"overall": 0.78, "breakdown": {...}, "feedback": "..."},
        {"overall": 0.76, "breakdown": {...}, "feedback": "..."}
    ],
    "attempts": 3
}
```

No outbound message is written to the `message` table in this case — the drafts
were rejected. The owner sees the three drafts + feedback in the HITL panel and
can pick one, edit one, or write their own. When the owner sends via HITL, a
`message` row is written with `role='human'`.

**Scores are stored in `message.metadata`** (on successful sends):
```json
{
  "eval_score": 0.91,
  "eval_attempts": 2,
  "eval_scores_breakdown": { "tone_match": 0.9, "personalisation": 0.95, ... },
  "angle_used": "Multiple reviews mention chronic understaffing...",
  "model": "gemini/gemini-3-flash-preview",
  "eval_model": "gemini/gemini-3.1-flash-lite-preview",
  "prompt_version": "generation-v1",
  "eval_prompt_version": "evaluation-v1",
  "tokens_used": 312
}
```

## 5. Reply Handling Pipeline

When a reply arrives via webhook, it goes through intent classification before any
response is drafted.

```
Webhook received → 202 Accepted within 2s (payload queued for background processing)
          ↓
[background task]
Resolve lead by (workspace_id, contact_uri)
          ↓
  (no match → write unmatched_webhook; done)
          ↓
Resolve target thread (most-recent-outbound rule if lead is in multiple campaigns)
          ↓
  (no active thread → write unmatched_webhook; done)
          ↓
Acquire thread lock (Postgres FOR UPDATE / SQLite BEGIN IMMEDIATE)
          ↓
Append message (role='lead')
          ↓
Re-read last N messages (now includes the just-appended inbound)
          ↓
Intent classification
          ↓
Route (see §5.1 routing table)
          ↓
      ┌──────────────────┬──────────────────────────────┬──────────────────┐
      │ Auto-reply       │ Escalation (paused_for_hitl) │ Close (won/lost) │
      │                  │                              │                  │
      │ Draft reply      │ Set thread.status =          │ Set statuses     │
      │ Self-evaluate    │   paused_for_hitl            │   (thread,       │
      │ Send             │ Write hitl_reason +          │    campaign_lead,│
      │ Append (role=ai) │   hitl_context               │    lead)         │
      │ auto_reply_count++│ Log HITL (v1: Web Push)     │ Log notification │
      └──────────────────┴──────────────────────────────┴──────────────────┘
          ↓
Release thread lock
```

In the POC, ingress ack uses FastAPI `BackgroundTasks` — the endpoint returns
202 immediately and processing runs in-process. In v1, the payload is pushed to
Redis and processed by Celery; the endpoint shape and `BaseConnector` interface
are identical in both tiers so the scale-up is a deployment change, not a code
change.

### 5.1 Intent Classification

A fast LLM call (using `eval_model` at temperature 0.0) that classifies the
incoming message and determines the next action.

**System prompt:**
```
You are classifying an incoming SMS reply to determine how an autonomous
outreach system should respond.

Campaign goal: {campaign_goal}
Message history (last 5 messages):
{recent_messages}

Incoming message: "{incoming_message}"

Classify the message and return a JSON object:
{
  "intent": one of:
    "positive"       -- interested, asking for more info, open to the goal
    "objection"      -- pushback, not right now, too busy
    "question"       -- specific question that needs a factual answer
    "negative"       -- clear no, unsubscribe, stop
    "unclear"        -- cannot determine intent with confidence
    "bot_check"      -- asking if this is automated or a bot
    "goal_achieved"  -- lead has agreed to the campaign goal
    "human_requested"-- lead explicitly asked to speak to a person,
  "requires_human": true | false,
  "confidence": 0.0–1.0,
  "reason": "one sentence explaining the classification"
}

requires_human must be true when:
- intent is "bot_check", "human_requested", or "goal_achieved"
- intent is "question" and confidence < 0.85
- intent is "unclear"
- confidence < 0.80 for any intent
```

**Routing table (evaluated top-to-bottom; first match wins):**

| Condition | Route | Statuses set |
|---|---|---|
| `intent == 'negative'` | **Close lost** — no reply | `thread='lost'`, `campaign_lead='lost'`, `lead='lost'` |
| `intent == 'goal_achieved'` | **Close won** — no reply; owner takes over | `thread='won'`, `campaign_lead='won'`, `lead='won'` |
| `intent == 'bot_check'` | **Escalate** — `hitl_reason='bot_check'` | `thread='paused_for_hitl'` |
| `intent == 'human_requested'` | **Escalate** — `hitl_reason='human_requested'` | `thread='paused_for_hitl'` |
| `intent == 'unclear'` | **Escalate** — `hitl_reason='unclear'` | `thread='paused_for_hitl'` |
| `confidence < 0.80` | **Escalate** — `hitl_reason='low_confidence'` | `thread='paused_for_hitl'` |
| `thread.auto_reply_count >= workspace.settings.max_auto_replies` | **Escalate** — `hitl_reason='max_auto_replies_reached'` | `thread='paused_for_hitl'` |
| Otherwise (`positive` / `objection` / `question` with high confidence under cap) | **Auto-reply** (§5.2) | `thread='active'`, `campaign_lead='replied'`, `lead='replied'`; `auto_reply_count++` on send |

**Status propagation rule:** When an inbound message closes a thread (`won` /
`lost`) or when an outbound reply is sent, the statuses on `campaign_lead` and
`lead` are updated in the same transaction as the `thread` update. The three
statuses are always consistent — there is no state where `thread.status='won'`
but `lead.status='contacted'`.

**HITL context for classifier escalations:** When escalating via the classifier
(not eval failure), `thread.hitl_context` stores the classifier output for the
owner's reference:

```json
{
  "intent": "unclear",
  "confidence": 0.61,
  "reason": "Message references a prior conversation we have no record of.",
  "incoming_message_id": "..."
}
```

### 5.2 Auto-Reply Generation

When the classification result routes to auto-reply, a new message is drafted using
the same generation and evaluation pipeline as the initial outreach, with two
differences:

1. The full message history (last 10 messages) is included in the generation prompt
   as conversation context.
2. The `angle` stored on the thread is reused — not regenerated.

**Reply generation system prompt additions:**
```
This is a follow-up reply in an ongoing conversation, not a first-contact message.

Conversation so far:
{message_history}

The lead's latest message: "{incoming_message}"
Classification: {intent} (confidence: {confidence})

Continue the conversation naturally. Address what the lead said directly.
Stay oriented toward the campaign goal without being pushy.
Keep the reply to 160 characters.
```

## 6. Connector Interface

The connector is the abstraction layer between AutoSDR's messaging logic and the
underlying communication channel. Adding a new connector (e.g. email) requires
implementing one class — nothing else changes.

### 6.1 Abstract Base Class

Connectors expose two inbound paths and one outbound path. `send` and
`parse_webhook` are abstract; `poll_incoming` has a no-op default so
webhook-only connectors remain valid without change.

```python
from abc import ABC, abstractmethod
from dataclasses import dataclass, field
from datetime import datetime, timezone
from typing import Any

@dataclass
class IncomingMessage:
    contact_uri: str                          # sender's phone number or email
    content: str                              # message body
    received_at: datetime = field(            # UTC timestamp
        default_factory=lambda: datetime.now(tz=timezone.utc))
    raw_payload: dict[str, Any] = field(default_factory=dict)
    provider_message_id: str | None = None    # used for dedupe across polls

@dataclass
class OutgoingMessage:
    contact_uri: str        # recipient phone number or email
    content: str            # message body

@dataclass
class SendResult:
    success: bool
    provider_message_id: str | None = None
    error: str | None = None

class BaseConnector(ABC):
    connector_type: str = "base"

    @abstractmethod
    async def send(self, message: OutgoingMessage) -> SendResult:
        """Deliver `message`. Raise ConnectorError on unrecoverable failure."""

    @abstractmethod
    def parse_webhook(self, payload: dict[str, Any]) -> IncomingMessage:
        """Normalise a raw provider webhook payload into IncomingMessage."""

    @abstractmethod
    async def validate_config(self) -> tuple[bool, str]:
        """Return (ok, detail) — e.g. ping the provider, check credentials."""

    async def poll_incoming(self) -> list[IncomingMessage]:
        """Pull unread inbound messages from the provider.

        Default is a no-op. Polling connectors (e.g. TextBee) override to
        fetch new messages since the last poll and are responsible for
        deduplication, so the scheduler can call this freely every tick.
        """
        return []
```

### 6.2 SMS Connectors (POC)

Three SMS connector implementations ship with AutoSDR. All three honour
the same `BaseConnector` interface, so campaigns, threads, and pipelines
do not care which is configured.

**File (`file`) — local harness.** Writes outbound to `data/outbox.jsonl`
and reads simulated inbound from `data/inbox.jsonl` (or via
`POST /api/webhooks/sim`). Used in tests and for end-to-end runs without
a phone. Default when no `CONNECTOR` env is set.

**TextBee (`textbee`) — hosted poll gateway.** Polls `GET
/api/v1/gateway/devices/{device_id}/get-received-sms` every
`INBOUND_POLL_S` seconds, tracks seen `smsId` values in-process for
dedupe, and sends via `POST /api/v1/gateway/devices/{device_id}/send-sms`.
Needs only an API key and device ID — **no public URL, no tunnel, no
Docker**. Fastest path to real SMS if you already have (or can stand up)
a TextBee-paired Android device.

```python
class TextBeeConnector(BaseConnector):
    connector_type = "textbee"

    def __init__(self, api_url, api_key, device_id, poll_limit=50):
        ...

    async def send(self, message: OutgoingMessage) -> SendResult: ...
    async def poll_incoming(self) -> list[IncomingMessage]: ...
    def parse_webhook(self, payload): ...  # usable if webhook path is enabled
    async def validate_config(self) -> tuple[bool, str]: ...
```

**SmsGate (`smsgate`) — self-hosted push gateway.** Wraps
[`capcom6/android-sms-gateway`](https://github.com/capcom6/android-sms-gateway).
Two deployment shapes are supported:

- **Private server mode.** `docker run -p 3000:3000 capcom6/sms-gateway-server`
  on the owner's machine; the Android app registers with that server and
  AutoSDR talks to the server over the LAN.
- **Local mode.** The Android app's own REST API, with AutoSDR on the
  same machine / LAN.

Outbound uses `POST {api_url}/messages` with HTTP Basic auth:

```python
{
    "phoneNumbers": [message.contact_uri],
    "textMessage": {"text": message.content},
}
```

Inbound arrives as a webhook `POST /api/webhooks/sms` with event
`sms:received`; other events (`sms:sent`, `sms:delivered`, …) are logged
and 202-accepted so the phone does not retry forever. Three env vars are
required — `SMSGATE_API_URL`, `SMSGATE_USERNAME`, `SMSGATE_PASSWORD` —
and `validate_config()` does a cheap authenticated `GET /messages` to
prove the server is reachable.

```python
class SmsGateConnector(BaseConnector):
    connector_type = "smsgate"

    def __init__(self, *, api_url, username, password, http_timeout_s=20.0):
        ...

    async def send(self, message: OutgoingMessage) -> SendResult: ...
    def parse_webhook(self, payload): ...                    # sms:received → IncomingMessage
    async def validate_config(self) -> tuple[bool, str]: ...  # GET /messages with basic auth
    # poll_incoming() is intentionally the no-op default — SmsGate pushes.
```

**Why ship both TextBee and SmsGate:** they hit different trade-offs —
TextBee is hosted and needs no tunnel but charges for volume; SmsGate is
fully open source, self-hosted, and push-native but needs a Docker
server (or LAN-reachable phone). The `BaseConnector` split means
switching between them is a one-line env change (`CONNECTOR=textbee` vs.
`CONNECTOR=smsgate`) with no pipeline changes.

**Adding httpSMS / v1 upgrade path:** the webhook endpoint used by
SmsGate (`/api/webhooks/sms`) is already connector-agnostic — it hands
the payload to `app.state.connector.parse_webhook()`. Swapping in an
httpSMS or webhook-only TextBee deployment is a `parse_webhook()`
implementation plus a tunnel URL; nothing upstream has to change.

### 6.3 Test Modes: Dry-run and Override

Two connector-level test modes ship with the POC. Both compose with
whatever `CONNECTOR=…` you have configured, so you do not need to
rewire real credentials to rehearse a run.

**Dry-run (`autosdr run --dry-run` or `DRY_RUN=true`).** Forces the
`FileConnector` regardless of `CONNECTOR`. The full generation,
self-evaluation, and intent-classification pipeline still runs against
the live Gemini credentials; every outbound "send" is written to
`data/outbox.jsonl` instead of hitting a real gateway. Use this to:

- Review what the agent would say before pointing it at a real phone.
- Regression-test prompt changes against a cohort of leads safely.
- Run CI / smoke tests without burning SMS quota or waking up a phone.

**Override (`autosdr run --override-to +61…` or
`SMS_OVERRIDE_TO=+61…`).** Wraps whichever connector is active
(`FileConnector` / `TextBeeConnector` / `SmsGateConnector`) in an
`OverrideConnector`. Every outbound message has its `contact_uri`
rewritten to the single override number before it hits the underlying
connector, and the wrapper remembers the last original recipient so
inbound messages coming back from the override number are remapped to
the correct lead/thread. Use this to:

- Smoke-test the real SMS path (gateway, carrier, formatting) against
  one phone you control, without blasting real leads.
- Run a scripted conversation end-to-end — outbound goes to your test
  phone, your replies get re-threaded against the lead the test covered.

Composing the two modes is explicit: `--dry-run --override-to +61…`
gives you the `OverrideConnector` wrapping the `FileConnector`, which is
a useful shape for debugging the override rewiring itself without any
real SMS traffic.

The factory enforces precedence so the observable behaviour is always
obvious from the settings:

```python
def get_connector(settings) -> BaseConnector:
    if settings.dry_run:
        inner = FileConnector(...)          # forced; CONNECTOR env is ignored
    elif settings.connector == "textbee":
        inner = TextBeeConnector(...)
    elif settings.connector == "smsgate":
        inner = SmsGateConnector(...)
    else:
        inner = FileConnector(...)

    if settings.sms_override_to:
        return OverrideConnector(inner, settings.sms_override_to)
    return inner
```

### 6.4 Adding Future Connectors

To add an email connector:

1. Implement `BaseConnector` in a new class `EmailConnector`.
2. Add `email` to the `connector_type` enum in the database.
3. Add the connector's API key fields to the workspace settings schema.
4. Register the webhook endpoint: `POST /api/webhooks/email/{workspace_id}`.

No changes to the generation pipeline, classification logic, or thread model are
required.

## 7. Prompt Versioning

Prompts are not stored in the database — they live in the codebase as versioned
constants. Each prompt module includes a version string that is written to
`message.metadata` at generation time.

```python
# autosdr/prompts/analysis.py
PROMPT_VERSION = "analysis-v1"
SYSTEM_PROMPT = """..."""
```

This allows regression testing: if a prompt change causes a drop in eval scores,
the version in `message.metadata` identifies exactly which messages were affected.

## 8. Evaluation Strategy

### 8.1 Automated (per-message)

Every generated message is scored by the self-evaluation step before sending.
Scores are stored in `message.metadata`. This provides a continuous quality signal
without any manual effort.

**Alert threshold:** If the rolling 7-day average eval score across a campaign drops
below 0.80, a warning is surfaced in the campaign dashboard. This may indicate a
tone prompt that needs recalibration or a lead list with too little data in
`raw_data` to generate strong angles.

### 8.2 HITL Routing Accuracy (periodic)

The intent classifier's routing decisions can be reviewed in the Messages section
of the frontend. Each escalated thread shows the classifier's output (`intent`,
`confidence`, `reason`) alongside what the owner actually did (replied, closed,
handed back to AI).

This creates a labelled dataset over time. At v1.0, a benchmarking script will
replay this dataset against the classifier to measure accuracy. Target: >= 90%
routing accuracy.

### 8.3 Outcome Tracking

Each `campaign_lead` has a final `status` (`won`, `lost`, `skipped`). Campaign
analytics show win rate, average thread length, and average time to close. These
are the business-level metrics the owner cares about most.
